Página 4 1. Introdução  
1.1. Propósito  
Este documento define a especificação técnica completa do banco de dados do CEOB 
(Comando do Exército Operacional Brasileiro) para a Fase 1 do projeto. O objetivo é 
estabelecer a estrutura fundacional que suportará todas as operações administrativas do 
sistema integrado entre Roblox, Discord e Railway.  
O banco de dados é a fonte única da verdade (Single Source of Truth) para todos os dados 
de militares, organizações militares, hierarquia e eventos do sistema.  
1.2. Escopo da Fase 1  
A Fase 1 contempla a model agem completa do schema PostgreSQL e a especificação técnica 
de todas as tabelas, relacionamentos, constraints e regras de negócio. Não inclui 
implementação de código (API, Bot) nem processos de promoção/transferência detalhados 
(Fase 2).  
 
Entregáveis da Fase 1:  
• Schema DDL completo para PostgreSQL  
• Definição de todas as tabelas, colunas, tipos e constraints  
• Enumerações (ENUMs) para domínios fechados  
• Relacionamentos e chaves estrangeiras  
• Regras de negócio e validações  
• Timeline genérica prepara da para todos os tipos de eventos  
• Estrutura organizacional dinâmica (OMs)  
• Fluxos de processo de registro e listagem  
1.3. Fases do Projeto  
Fase Escopo  Status

Página 5 2. Arquitetura Geral  
2.1. Visão Geral da Integração  
O sistema opera em três camadas integradas pelo Railway. O PostgreSQL armazena todos 
os dados e é a fonte da verdade. O Bot do Discord é a única interface de alteração de dados. 
O Roblox fornece identificação de jogadores e a tualizações de grupo.  
 
Fluxo de dados:  
• Roblox → fornece User ID do jogador e dados do grupo  
• Discord Bot → único ponto de entrada para alterações no banco  
• PostgreSQL → armazena, valida e persiste todos os dados  
• Boletim Eletrônico → publicação automática no Discord após cada alteração  
• API Pública → consulta de ficha (somente leitura)  
2.2. Princípios Arquiteturais  
• Banco como fonte da verdade: Toda informação autoritária reside no PostgreSQL. 
Discord e Roblox são consumidores.  
• Imutabilidade de histórico: Registros na timeline nunca são editados ou excluídos. 
Correções geram novos eventos.  
• Soft delete: Nenhum registro é fisicamente excluído. Utiliza -se flags de status 
(ativo/desativado).  
• Auditoria completa: Toda alteração registra quem fez, quando e por quê . 
• Extensibilidade: O schema é projetado para acomodar as Fases 2 e 3 sem 
migrações destrutivas.  
2.3. Estrutura Organizacional do CEOB  
A estrutura hierárquica do CEOB é dinâmica, permitindo criação e desativação de 
Organizações Militares (OMs) ao longo do t empo. A árvore organizacional é modelada com 
auto-referência (parent_id).

Geração da matrícula:  
A matrícula é gerada automaticamente via função ou trigger no PostgreSQL. O formato é 
YYYY -NNNN onde YYYY é o ano de ingresso e NNNN  é um sequencial reiniciado a cada 
ano. Exemplo: 2026 -0001, 2026 -0002, ..., 2027 -0001.

Página 19 6. Regras de Negócio  
6.1. Regras de Integridade  
• RN-001: A matrícula é gerada automaticamen te e imutável após criação. Formato: 
YYYY -NNNN.  
• RN-002: Um militar deve ter exatamente um roblox_user_id único. Não pode haver 
dois militares com o mesmo Roblox ID.  
• RN-003: Um militar deve estar lotado em exatamente uma OM a qualquer momento 
(om_lotacao_id  NOT NULL).  
• RN-004: Um militar deve possuir exatamente uma patente a qualquer momento 
(patente_id NOT NULL).  
• RN-005: Registros na timeline_eventos são append -only. Nunca podem ser 
alterados ou excluídos.  
• RN-006: Toda alteração no banco deve ser publicada e m boletim eletrônico 
automático.  
• RN-007: Alterações são feitas exclusivamente pelo Bot do Discord. Acesso direto ao 
banco é restrito a administradores.  
6.2. Regras de Hierarquia  
• RN-008: Todo militar com patente >= Coronel (ordem_precedencia <= 5) é 
automat icamente integrante do Alto Comando.  
• RN-009: O Comando Supremo aceita no máximo 3 militares.  
• RN-010: Um requerimento de registro só pode ser solicitado por militar com patente 
> Cabo.  
6.3. Regras de Funções  
• RN-011: Funções são cumulativas: um militar pode exercer múltiplas funções em 
OMs diferentes simultaneamente.  
• RN-012: A mesma função na mesma OM não pode ser duplicada para o mesmo 
militar enquanto ativa.  
• RN-013: Ao encerrar uma função, o campo data_fim é preenchido e ativo é setado 
para false. O registro nunca é excluído.  
6.4. Regras de Situação Funcional  
• RN-014: Transições de situação funcional devem respeitar o fluxo: ATIVO pode ir 
para qualquer estado. EXCLUIDO  e REFORMADO são estados finais.  
• RN-015: Toda mudança de situação gera evento MUDANCA_SITUACAO na 
timeline.  
6.5. Regras de Requerimento e Listagem  
• RN-016: Listagem é processo interno do órgão responsável. O militar alvo não tem 
acesso.  
• RN-017: Requerimento passa obrigatoriamente por validação automática e análise 
de mérito pela DGP.  
• RN-018: Após aprovação: publica no BI eletrônico, notifica o militar, regist ra na 
timeline.

Página 21 7. Fluxos de Processo (Fase 1)  
7.1. Requerimento de Registro  
Fluxo iniciado quando um militar com patente superior a Cabo solicita o registro de um novo 
ingressante.  
 
Passos : 
• 1. Militar com patente > Cabo inicia o requerimento via Bot do Discord.  
• 2. Sistema realiza validação automática (patente do solicitante, dados do 
ingressante).  
• 3. DGP recebe para análise de mérito.  
• 4a. Se APROVADO: publica no BI eletrônico → notifica → r egistra na timeline 
(INGRESSO) → cria o registro do militar.  
• 4b. Se INDEFERIDO: registra motivo → notifica o solicitante.  
 
Registros gerados (aprovado):  
• INSERT em ceob.militares (novo militar)  
• INSERT em ceob.timeline_eventos (tipo: INGRESSO)  
• INSERT em ceob .postagens (vinculada ao solicitante)  
• INSERT em ceob.boletim_eletronico  
• INSERT em ceob.requerimentos (com status APROVADO)  
7.2. Listagem de Registro  
Fluxo iniciado internamente pela DGP, sem solicitação externa. O DGP verifica necessidade 
de efetivo e even tuais requisições especiais do ingressante.  
 
Passos:  
• 1. Membro da DGP inicia formulário padrão via Bot.  
• 2. DGP verifica necessidade de efetivo nas OMs e requisição especial.  
• 3. Publica no BI eletrônico → notifica → registra na timeline.  
 
Critérios para decisão da OM:  
• Necessidade de efetivo (dados de todas as OMs)  
• Requisição especial do registrado (ex: Voluntário na unidade PQD)
